# 散歩判（sanpo-ban）

個人向けダイエット情報ダッシュボード。散歩・食事（PFC）・筋トレ・トレッドミル・体重を一か所に集約し、iPhone の歩数を自動取り込みする。家計簿型の入力負荷は避け、ガチダイエットに必要な数値は追える。

## 目的

- ダイエットに必要な情報（摂取・消費・体重・運動）を単一アプリに集約する
- 記録の入力負荷を下げつつ、カロリー・PFC・歩数は正確に把握する
- 散歩の「ダイエット」と「発見の楽しさ」を両立して記録する

## スコープ

- 含む: Web UI、REST API、MySQL、iPhone Health 同期（ショートカット）、食事・運動・体重管理、週次サマリー
- 含まない: ユーザー認証・OAuth、会社業務連携、SNS 自動連携、クラウドホスティング（初期）

## イントークで確認した事項

### ユーザー像

- 会社員 SE、個人利用（仕事への自作アプリ導入は消極的）
- iPhone 利用、散歩時はスマホを携行
- ダイエットを本気で継続したいが、記録入力はだるくて未実施だった

### 機能ニーズ

- カロリー計算、PFC（取れるなら）、歩数（自動）、体重、筋トレ、ジムのトレッドミル（手入力）
- 食事は「食べていいか迷う」→ プリセット・バーコード・手入力の併用
- 散歩はダイエット目的＋街の発見メモ（任意）
- ゲーム・一人旅は Phase 2 以降（旅モード・Insta タグメモ）

### 技術・運用方針

- **DB:** MySQL
- **進め方 A:** PC ローカルでプロトタイプ → Raspberry Pi へ移行
- **認証:** なし（自宅 LAN のみ）
- **ホスティング:** 最終的に Raspberry Pi。SD カード寿命対策として USB SSD 推奨（データ at minimum、可能なら OS も SSD 起動）
- **外部 API:** Open Food Facts（OSS、バーコード時）。OAuth 系（Fitbit 等）は使わない

### 未決（要件定義で整理）

- 目標 kcal/PFC の決め方（自動 TDEE vs 手入力）→ 両方サポート、初回は自動提案＋ユーザー確定を仮定
- 筋トレ詳細（セット・重量）の Phase 1 範囲 → クイック必須、詳細は Phase 2
